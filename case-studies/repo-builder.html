repo_builder — an AI agent orchestration platform

A harness-agnostic platform that does deterministic orchestration of non-deterministic AI agents: humans, cron, and webhooks compose durable workflows (plan → build → review → fix) whose step order survives a crash, while the intelligent work inside each step is delegated to a swappable, supervised external AI harness — Claude Code and pi ship today.

01The problem

AI coding agents are powerful but non-deterministic. If you let an agent decide the whole shape of a multi-step job — plan, then build, then review, then fix — the control flow becomes as unpredictable as the model, and a crash mid-run loses everything. For real software-delivery work you want the opposite: the sequence of steps and the branching should be fixed, auditable, and crash-durable, while only the intelligence inside each step stays probabilistic.

A second trap is lock-in. Most agent tooling is welded to one vendor's CLI or SDK. The moment a better model or harness appears, you rewrite the orchestration layer.

02The spec I built to

I built repo_builder spec-first, the same way I work in Claude Code: the desired behaviour is written down as explicit, checkable requirements before implementation, and the spec — not a prompt — is the source of truth. The load-bearing requirements were:

  • Deterministic, durable workflows. Every state transition is persisted before it happens, so an in-flight run resumes from its current step after a node restart.
  • Harness-agnostic by construction. Adding a new AI harness must be one adapter module + one config entry — zero edits to the core event types, schema, or runtime.
  • Typed throughout. A @spec on every public function, tagged {:ok, t()} | {:error, reason()} tuples over raising, and a CI gate that fails on a single warning.
  • Safe by default. No orphaned child processes, no secrets in the database, signed external triggers only.

03Architecture — a deterministic core driving non-deterministic agents

The core is a deterministic state machine; the intelligence in each step is an external CLI spoken to over a normalized event stream. Two execution paths share one canonical contract: a live path (a WorkflowEngine.Runner feeding a real-time dashboard) and a durable path (Oban workers that persist each transition and reconcile in-flight runs after a restart).

The seam that makes harness-swapping cheap is a single closed type: RepoBuilder.Harness.Event, an 8-variant typedstruct sum type carrying an open harness :: atom() identity plus a raw escape hatch. Because the identity is open, a new harness needs no change to the event types — only a two-callback adapter (command/1 + normalize/2) behind a registry seam. A bundled no-op Cursor adapter proves a third harness drops in with zero core edits.

Each live agent runs as one erlexec-backed GenServer child with multibyte-safe NDJSON buffering, idle timeout, stdout-overflow backpressure, and an admission concurrency gate — classic OTP supervision applied to external AI processes.

04Reliability & guardrails

This is where the Elixir/OTP background earns its place — the failure modes of spawning many long-running external processes are exactly what BEAM supervision is built for:

  • Zero-orphan guarantee. A durable OS-pid ledger plus a boot-time OrphanReaper that verifies /proc/<pid>/environ before signalling, so a crash never leaks runaway agent processes.
  • Secret-safe persistence. Events are redacted before they hit the database; the live broadcast keeps full detail.
  • Authenticated triggers. Webhooks are verified with constant-time HMAC-SHA256 over "<timestamp>.<raw_body>" plus a freshness window — rejected before any job is enqueued, so a bad request never becomes a poison job.
  • Cost awareness. Per-run cost is streamed and tracked, with alerting above a configurable threshold; unpriced models report nil rather than a misleading 0.0.

05Extensibility — point it at any repo, extend it with plugins

The platform can target any repository, not just itself: a target repo is promoted to a first-class Project with an auto-detected stack and capability map, a pinned command pack, a budget cap, and optional git worktree isolation so parallel agents never collide. A stack-aware resolver fills capability tokens ({{TEST_COMMAND}} …) so one command body adapts to every stack.

The same "add a harness = one module + one config entry" doctrine generalizes to a plugin system: versioned packages contribute to a closed set of extension points (command packs, data-defined workflow types, agent templates, orchestrator context fragments, capability data, and code-layer harness adapters) with an open string identity, installable from a local library or a remote store and activated per project.

06Validation

The whole system is verified the way I want production software verified — not by eyeballing agent output:

  • 146 automated tests drive the runtime through a Fake/Mock adapter via the registry seam, so the suite needs no external CLIs and runs deterministically in CI.
  • A five-command green gate — mix compile --warnings-as-errors, format --check-formatted, credo --strict, test --warnings-as-errors, dialyzer — enforced locally and in GitHub Actions.
  • The only manual step is live acceptance against the real claude and pi CLIs, since those are external tools.

07What it taught me

The lesson that transfers to any agentic role: the leverage is in the boundary, not the model. A small, closed, well-typed event contract with one open field bought harness-agnosticism, testability without live CLIs, and crash-durability all at once. Designing that seam — and writing the spec that demanded it — mattered more than any single prompt.
